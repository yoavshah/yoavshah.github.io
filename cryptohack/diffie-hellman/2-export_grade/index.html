Challenge

In this challenge we are an attacker sits between Alice and Bob and we perform “Man in the Middle” attack. We cannot change any parameter but the number of bytes chosen and support in the algorithm. Of course we will choose DH-64 bits, but how can we crack it?

Solution

This was quite a hard challenge because I wanted to understand the algorithms used to expolit DH with small keys. First I assume you understand the Diffie-Hellman algorithm and the Discrete-Log problem (you can visit my other post if not).

First we will make Alice and Bob use DH-64 bit, which uses small Diffie-Hellman prime number, and we will use this to solve the discrete-log problem resulting in finding the shared_secret.

Chinese Reminder Theorem

First I will talk about the Chinese Reminder Theorem. I have watched this awesome video that explains it well

In this algorithm we get a list of tuples in length of $l$ ($r_i$, $p_i$) and we need to find $x$ such that

$$ x \equiv_{p_1} r_1 $$$$ x \equiv_{p_2} r_2 $$$$ x \equiv_{p_3} r_3 $$$$ … $$

To solve this problem we need first to calculate the multiplication of $M = p_1 * p_2 * p_3 * … $

then iterate over each equation we got in the input list and calculate

$$ t_i = M / p_i $$

So we get that for each $j \neq i$ $$ t_i \equiv_{p_j} 0 $$

and we also get that $$ t_i \not\equiv_{p_i} 0 $$

Now we need to find $s_i$ such that $$ t_i * s_i \equiv_{p_i} 1 $$

Then we can multiply by $r_i$ the equation to get $$ r_i * t_i * s_i \equiv_{p_i} r_i $$

We can iterate over $s_i$ in the field to find it. Also, we can see that for each $j \neq i$ $$ r_i * t_i * s_i \equiv_{p_j} 0 $$

With it we can see that $$ r_0 * t_0 * s_0 + r_1 * t_1 * s_1 + … \equiv_{p_0} r_0 $$$$ r_0 * t_0 * s_0 + r_1 * t_1 * s_1 + … \equiv_{p_1} r_1 $$$$ … $$

So we found such an $x$. We can now make it smaller by modding it with $M$.

Pohlig-Hellman

First I found the algorithm “Pohlig-Hellman” (This video explains this algorithm great).

Let’s say we want to solve the following discrete log equation

$$ log_a(b) \bmod p $$

First have a look at the Euler’s totient function.

Because p is prime then $\varphi(p) = (p-1)$. Let’s say $p-1 = p_1 * p_2$ (I use here 2 factors for simplicity but it will work also for more).

We need to find $x$ such that $$ a^x \equiv_p b $$

Let $x = a_0 + a_1*p_1$

So we got $$ a^{a_0 + a_1*p_1} \equiv_p b $$

Now we can power up both sides by $p_2$ (with more factors we can just power it up with their multiplication), so we get

$$ a^{p_2(a_0 + a_1*p_1)} \equiv_p b^{p_2} $$

So $$ a^{a_0p_2} * a^{(a_1p_1*p_2)} \equiv_p b^{p_2} $$

$$ (a^{p_2})^{a_0} * (a^{a_1})^{(p_1*p_2)} \equiv_p b^{p_2} $$

$$ (a^{p_2})^{a_0} * (a^{a_1})^{\varphi(p)} \equiv_p b^{p_2} $$

Using the Euler-Fermat Theorem

$$ x^{\varphi(n)} \equiv_n 1 $$

We can remove the second part in the equation, so $$ (a^{p_2})^{a_0} \equiv_p b^{p_2} $$

We know $p2$, $a$, $b$ so the only variable we do not know is $a_0$. Because $x = a_0 + a_1 * p_1$, then $a_0$ will be some value between $0$ to $p_1$.

It is easier than the original discrete-log equation because the original solution could range from $0$ to $p$, whether the new equation’s solution could range only from $0$ to $p_1$ - which is smaller than $p$.

Then using trial-and-error we can solve this new easier discrete log equation.

Using the same algorithm we iterate over all the factors of $\varphi(p)$, and calculate its discrete log.

After that all of the iterations we get $x \equiv_{p_1} a_0$ $x \equiv_{p_2} b_0$ $x \equiv_{p_3} c_0$ …

And we can find $x$ using the Chinese Reminder Theorem I talked about above to find $x$!

So I have wrote this algorithm in the following file pohlig_hellman.py

Then I tried to use this code to solve the 64 bit discrete-log equation, but one of the prime factor of $\varphi(p)$ was too large and it took too much time to iterate over all the possible values, so I just tried another algorithm I found.

Baby Step Giant Step (BSGS)

This algorithm is another algorithm to solve the discrete-log problem. This algorithm is based on a space-time tradeoff, meaning that instead of bruteforcing using trial and error N possible values, we store x values and compare them againts $N / x$ values.

So if we choose $sqrt(N)$ we get that it will run much faster than the bruteforce method, but will require much more space.

This is a great video about this algorithm

The algorithm works as follows, given a, b, p we need to find

$$ log_a(b) \bmod p $$

Or calculate $x$ such that

$$ a^x \equiv_p b $$

First let’s choose $K = \lceil{sqrt(p)}\rceil$, but we can choose any other $K$ we wish.

We will iterate over $K$ and calculate for each $$ a^i \bmod p $$ and store it in hashtable.

Then we will iterate another $K$ times and each time will calculate $$ b * a^{-K * i} $$ each iteration we will check if the current value is inside the hashtable, when we find a match we can use the hashtable to find j such that $a^j \bmod p = b * a^{-K * i}$, so

$$ a^j \bmod p \equiv_p b * a^{-K * i} $$

$$ a^j * a^{K * i} \bmod p \equiv_p b $$

$$ a^{j + K * i} \bmod p \equiv_p b $$

So the solution to the discrete log equation is $j + K*i$

I created my version of this algorithm but it still was very slow, the $sqrt(p)$ was too large and iterating through it will require better hardware.

Then I stepped uppon a combined algorithm of both Pohlig-Hellman and Baby Step Giant Step.

Overall Solution

I found out that I can use Pohlig-Hellman and instead of bruteforcing the easier discrete-log equations, I could use the Baby Step Giant Step algorithm!

So after performing Pohlig-Hellman, I found a list of mods and their reminders that I had to send to the Chinese Reminder Theorem, the next problem rose in my implementation.

In my implementation the code bruteforce a value to find the inverse of a number in a field.

So I create a new implementation which for every mod the input will send whether the mod is prime or not, if it is prime then we can find the inverse using Fermat’s Little Theorem, which conclude

$$ a^p \equiv_p a $$ So $$ a^{p-1} \equiv_p 1 $$ So $$ a^{p-2} \equiv_p a^{-1} $$

Now I could find the inverse easly and thus making it more efficient! (There is actually a way to find the inverse efficiently - extended gcd, but I didn’t need this right now)

from pwn import *

#from helpers import pohlig_hellman
from helpers import pohlig_hellman_and_bsgs


def is_pkcs7_padded(message):
    padding = message[-message[-1]:]
    return all(padding[i] == len(padding) for i in range(0, len(padding)))


def decrypt_flag(shared_secret: int, iv: str, ciphertext: str):
    # Derive AES key from shared secret
    sha1 = hashlib.sha1()
    sha1.update(str(shared_secret).encode('ascii'))
    key = sha1.digest()[:16]
    # Decrypt flag
    ciphertext = bytes.fromhex(ciphertext)
    iv = bytes.fromhex(iv)
    cipher = AES.new(key, AES.MODE_CBC, iv)
    plaintext = cipher.decrypt(ciphertext)

    if is_pkcs7_padded(plaintext):
        return unpad(plaintext, 16).decode('ascii')
    else:
        return plaintext.decode('ascii')
    
def get_json(s):
    return json.loads(s[s.index(b"{"):])


r = remote('socket.cryptohack.org', 13379, level = 'debug')

_ = r.recvline()
_ = r.recv()

r.sendline(json.dumps({"supported": ["DH64"]}).encode())

_ = r.recvline()
_ = r.recv()

r.sendline(json.dumps({"chosen": "DH64"}).encode())

alice_dh_params = get_json(r.recvline())
bob_dh_params = get_json(r.recvline())
enc_flag = get_json(r.recvline())

p = int(alice_dh_params["p"], 16)
g = int(alice_dh_params["g"], 16)
A = int(alice_dh_params["A"], 16)
B = int(bob_dh_params["B"], 16)

r.close()


# g ** a = A mod p => find a
found_a = pohlig_hellman_and_bsgs.discrete_log(g, A, p)

# calculate shared secret
# B ** found_a mod p 
shared_secret = pow(B, found_a, p)

# decrypt flag
dec_flag = decrypt_flag(shared_secret, enc_flag["iv"], enc_flag["encrypted_flag"])

print(dec_flag)

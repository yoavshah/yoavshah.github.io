Reverse dbutil_2_3

Here I will show the reversing process of dbutil_2_3.sys (CVE-2021-21551)

DriverEntry

First have a look at the DriverEntry which is the first function that executes when the driver is loaded

As we can see, all this function does is to do some sanity checks and then execute sub_11008, let’s rename it to RealDriverEntry using the n keyword.

This is the RealDriverEntry function

Here we can see the device assosiated with the driver is being created (with undefined priviledges) with the "\Device\DBUtil_2_3" name and with with the symbolic link "\DosDevices\DBUtil_2_3".

We can see the MajorFunctions being set to the same function sub_11170, let’s call it GenericMajor.

We can see the initialization of the DeviceExtension of the driver, and that at DeviceExtension is a struct that at index 24 it is a PRKDPC object.

Let’s create a struct for our DeviceExtension and fill its fields one by one as we investigate this driver.

Go to the Structures window, press insert and call this struct dbutil_device_extension, now press a and create an array of size 24 bytes (we will use 24 for now because we don’t know the real size of the struct and we just want to fill bytes), press d to create a new field, press y to change its type to KDPC (in the decompiler we can see the pointer is PRKDPC so the field type is KDPC), and at last rename the field to something like kdpc_object.

Now press y on the DeviceExtension and set its type to dbutil_device_extension*.

You should have something like

Now let’s have a look at the GenericMajor function, first let’s set its type to PDRIVER_DISPATCH by pressing y on this function and writing it.

GenericMajor

This function is a big one so I will not be able to show an image with all of its code.

We can see here that this function also uses the DeviceExtension, so we can just change its type.

Also at line 30 there is an if that checks if the MajorFunction is 14, we can press the m keyword to set this value as an enum, just search for IRP_MJ_DEVICE_CONTROL.

Then we can se at line 33 that there is a mistake, it is setting field_0 at in the device extension struct to MasterIrp.

*(_QWORD *)DeviceExtension->field_0 = Irp->AssociatedIrp.MasterIrp;

We know that the IRP struct contains unions, and MasterIrp is not the correct one, we can press the keyword alt+y and set it to SystemBuffer which is the buffer received in the IOCTL. The same problem occurs many times, also at line 34, there the Create field is invalid and should be changed to DeviceIoControl.

After we changed them we can see that the first QWORD of DeviceExtension is a pointer to the SystemBuffer and the DWORD after it is the length of the input buffer, so let’s add it to the struct.

Now we can follow the control flow based on the IOCTL code and find out about some of the functionality this driver performs.

Here is a list of all the existing IOCTL codes in this driver:

0x9B0C1EC0

As we can see, this IOCTL just executes the following function sub_151D4

v15 = sub_151D4(DeviceExtension);

Let’s call it ioctl_9B0C1EC0

Now inside the function we know that the parameter sent is DeviceExtension, so let’s change its type to it.

We can see that there is the following check

if ( DeviceExtension->InputBufferLength != 48 )
    return 3221225485i64;

Then we can see a memmove operation from the SystemBuffer into an array of size 0x30. So let’s create a new struct that will represent this IOCTL data (let’s call it ioctl_9B0C1EC0_struct). Then we can set the local variable called Dst to the be of this type.

Now we can see this code which means that the &field_0[4] is actualy a QWORD, so I added an unknown field and then set this field to be just a random name I will notice next time it appear.

v3 = *(_QWORD *)&DeviceExtension->field_0[4];

After that we can see this compare

if ( unknown_address && unknown_address != *(_QWORD *)Dst.field_0 )
    return 3221225477i64;

We can conclude that the first pointer is of type QWORD.

After we changed the struct, we can see operations on Dst.field_0[8] and Dst.field_0[16] which are both QWORDS.

Now we see this call

ContiguousMemorySpecifyCache = MmAllocateContiguousMemorySpecifyCache(
                                LODWORD(Dst.field_8),
                                (PHYSICAL_ADDRESS)v4,
                                v5,
                                v6,
                                MmNonCached);

So field_8 is NumberOfBytes and v4 which is field_10 is LowestAcceptableAddress, so let’s rewrite the struct.

Then we can see the output goes into field_0 (remember it is not the real field_0 but the one after we changed the struct, it is 0x20 offset)

*(_QWORD *)Dst.field_0 = ContiguousMemorySpecifyCache;

So we can set this field as VirtualMemoryAllocated.

We can see then these code lines

PhysicalAddress = MmGetPhysicalAddress(ContiguousMemorySpecifyCache);
v9 = ((__int64 (__fastcall *)(_QWORD))sub_11574)((PHYSICAL_ADDRESS)PhysicalAddress.QuadPart);
*(_QWORD *)Dst.field_0 = v9

Which gets the newly created PhysicalAddress, then execute a little function with it that convert it to an __int64 number, and then set it to offset 0x28, we can change the struct.

unsigned __int64 __fastcall sub_11574(__int64 a1)
{
  return (unsigned int)a1 ^ (unsigned __int64)((__int64)SHIDWORD(a1) << 32);
}

Nice! So we successfully built the struct

00000000 ioctl_9B0C1EC0_struct struc ; (sizeof=0x30, mappedto_379)
00000000                                         ; XREF: ioctl_9B0C1EC0/r
00000000 unknown_address dq ?                    ; XREF: ioctl_9B0C1EC0+35/r
00000008 NumberOfBytes   dq ?                    ; XREF: ioctl_9B0C1EC0+64/r
00000010 LowestPhysicalAddress dq ?
00000018 field_18        dq ?
00000020 VirtualMemoryAllocated dq ?
00000028 PhysicalAddressAllocated dq ?
00000030 ioctl_9B0C1EC0_struct ends
00000030

Then the function just use the memmove from the local variable Dst to the DeviceExtension->SystemBuffer which will then returned to the user.

So an user without permissions can use this IOCTL to allocate NonCached contiguous memory! doesn’t seem right.

0x9B0C1EC4 + 0x9B0C1EC8

In this code we first set a variable to 1

LOBYTE(v8) = 1;

and then this function is execute - let’s call this function ioctl_9B0C1EC4 dive in

v15 = sub_15294(DeviceExtension, v8);

let’s fix the parameters types so we have clear view on what this function do

InputBufferLength = DeviceExtension->InputBufferLength;
if ( InputBufferLength < 0x18 )
    return 3221225485i64;

First we can see the input for the function is 0x18 bytes.

We can then see these lines of code

v11 = *(_QWORD *)DeviceExtension->SystemBuffer;
v12 = *(_QWORD *)(DeviceExtension->SystemBuffer + 8);
v13 = *(_QWORD *)(DeviceExtension->SystemBuffer + 16);

It seems that the the input from the user is 3 pointers.

v7 = InputBufferLength - 24;
v8 = (void *)(v12 + (unsigned int)v13);
if ( a2 )
{
    v9 = (const void *)(v12 + (unsigned int)v13);
    v8 = (void *)(SystemBuffer + 24);
}
else
{
    v9 = (const void *)(SystemBuffer + 24);
}
memmove(v8, v9, totalNumberMinus24);

We can see that v7 is actually the length minus 24 (which is the size of the 3 first pointers). Then we see some variables being set, let’s see their usage to understand what they are up to. They are using memmove from v9 to v8 using the remaining bytes available from the SystemBuffer, name them src_pointer and dst_pointer accordinly

When a2 is set, src_pointer equals v12 + v13 and dst_pointer equals to data at SystemBuffer which is later returned to the user, this is read operation

When a2 is not set, src_pointer equals to some index in SystemBuffer which received from the user, and dst_pointer equals to v12 + v13, this is write operation

So this function allows a user to read\write data based on a2, in this ioctl a2 is set to 1, meaning this ioctl allows to read any memory from the kernel.

After that we can see in the code path of IOCTL 0x9B0C1EC8 that it calls the same function but with a2 not set, so this is a write operation.

Those 2 IOCTL allows read and write operation on any address in the kernel! In the next article we will exploit this to elevate priviledges.

Summary

This was an example for reverse engineering a vulnerable driver, this driver has more nice IOCTLS to view but using the same methodology shown here you can reverse the other IOCTLs. In the next article I will show how to exploit the vulnerabilities shown here.
